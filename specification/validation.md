# 数字资产验证

数字资产验证是确认资产完整性、合规性和一致性的独立过程，与发现过程解耦但协作运行。

## 验证与发现的关系

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   发现过程   │ --> │   资产注册   │ --> │   验证过程   │
│  (Discovery)│     │  (Registry) │     │ (Validation)│
└─────────────┘     └─────────────┘     └─────────────┘
       │                                        │
       │         可选：发现时触发验证              │
       └----------------------------------------┘
```

**关键区别**：

| 维度 | 发现 (Discovery) | 验证 (Validation) |
|------|------------------|-------------------|
| 核心问题 | 「有什么？」 | 「对吗？」 |
| 触发时机 | 文件变更、初始化、定时 | 提交前、门禁检查、手动 |
| 执行频率 | 高频、自动 | 低频、按需 |
| 失败处理 | 记录差异、等待同步 | 阻断流程、要求修复 |

## 验证类型

### 1. 结构验证

验证资产是否符合契约定义的 schema：

- 必填字段是否缺失
- 字段类型是否正确
- 路径格式是否合法

### 2. 存在性验证

验证契约声明的资产在文件系统中真实存在：

- 文件/目录是否存在
- 符号链接是否有效
- 权限是否可访问

### 3. 一致性验证

验证契约与实际文件系统的一致性：

- 契约有但文件缺失（missing）
- 文件存在但契约未声明（unexpected）
- 双方定义冲突（conflict）

### 4. 内容验证

验证资产内容的合规性（可选）：

- 文档格式是否符合标准
- 代码是否通过 lint
- 元数据是否完整

## 验证流程

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  加载契约    │ -> │  扫描资产    │ -> │  执行检查    │ -> │  生成报告    │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

## 验证事件

| 事件 | 描述 |
|------|------|
| `validation.started` | 验证过程开始 |
| `validation.passed` | 验证通过 |
| `validation.failed` | 验证失败 |
| `validation.missing` | 发现缺失资产 |
| `validation.unexpected` | 发现未声明资产 |

## 验证配置

验证配置统一放在 `.quanttide/asset/contract.yaml` 中，与发现配置一起构成完整的治理配置。

```yaml
# .quanttide/asset/contract.yaml
spec_version: 0.0.1
version: 1.0.0

# 资产清单
assets:
  # ... 资产定义

# 发现配置（可选）
discovery:
  # ... 发现配置

# 验证配置（可选）
validation:
  # 验证级别
  level: strict  # strict | lenient

  # 验证规则
  rules:
    structure: true      # 结构验证
    existence: true      # 存在性验证
    consistency: true    # 一致性验证
    content: false       # 内容验证（可选）

  # 严格模式配置
  strict:
    fail_on_missing: true
    fail_on_unexpected: true
    fail_on_conflict: true

  # 与发现的集成
  auto_on_discovery: false  # 发现后是否自动验证
```

### 配置与契约的关系

```
┌─────────────────────────────────────────┐
│      .quanttide/asset/contract.yaml     │
│                                         │
│  ┌─────────┐ ┌───────────┐ ┌─────────┐ │
│  │  assets │ │ discovery │ │validation│ │
│  │(是什么) │ │(怎么发现) │ │(怎么验证)│ │
│  └─────────┘ └───────────┘ └─────────┘ │
│       │            │            │       │
│       └────────────┼────────────┘       │
│                    v                    │
│            ┌─────────────┐              │
│            │  治理引擎    │              │
│            └─────────────┘              │
└─────────────────────────────────────────┘
```

单一配置文件符合 Data Contract 理念：契约不仅定义资产结构，也定义资产的发现方式和验证规则，形成完整的治理规范。

## 与发现的协作模式

### 模式一：完全独立（推荐）

发现和验证各自独立运行，通过资产注册中心解耦：

- 发现只管「找到资产」
- 验证只管「检查资产」
- 用户/系统按需调用验证

### 模式二：发现触发验证

发现完成后自动触发验证：

```yaml
discovery:
  auto_validate: true  # 发现后自动验证
```

适用于对一致性要求严格的场景。

### 模式三：验证驱动发现

验证失败时触发发现，修复差异：

```
验证失败（发现 missing/unexpected）
    |
    v
触发发现过程重新扫描
    |
    v
生成修复建议或自动修复
```

适用于自动化修复场景。
