# Catalog 与 Contract 的区别

在数字资产管理中，**Catalog（目录）**和**Contract（契约）**是两个相关但不同的概念。

## 核心区别

| 维度 | Contract（契约） | Catalog（目录） |
|------|-----------------|----------------|
| **定义** | 项目应该有什么资产 | 特定时间点实际有哪些资产 |
| **性质** | 声明式、规范性 | 记录式、快照性 |
| **维护者** | 项目开发者 | 系统自动生成 + 人工触发快照 |
| **存储位置** | `.quanttide/asset/contract.yaml` | `.quanttide/catalog/<timestamp>.yaml` |
| **版本控制** | 是（Git 管理） | 是（快照文件版本控制） |
| **作用** | 约定资产组成 | 记录资产历史状态 |

## 类比理解

| 类比 | Contract | Catalog |
|------|----------|---------|
| **图书馆** | 图书采购清单 | 年度馆藏盘点报告 |
| **建筑** | 建筑设计图纸 | 竣工验收清单 |
| **API** | OpenAPI 规范 | API 版本发布清单 |
| **Data Contract** | 数据契约定义 | 数据资产快照 |
| **财务** | 预算计划 | 年度审计报告 |

## 关系图示

```
┌─────────────────┐         ┌─────────────────────────────────┐
│    Contract     │         │            Catalog              │
│   （契约/图纸）  │  ---->  │         （快照/盘点）            │
│  项目应该有什么  │  驱动    │    特定时间点实际有什么          │
└─────────────────┘         └─────────────────────────────────┘
         │                           │
         │                    ┌──────┴──────┐
         │                    │  发现/验证   │
         │                    │  生成快照    │
         │                    └─────────────┘
         v                           v
┌─────────────────┐         ┌─────────────────────────────────┐
│   人工编辑维护   │         │  自动生成 + 版本控制（快照文件）  │
│  版本控制追踪   │         │  里程碑归档、审计追溯            │

```

## 内容对比

### Contract 示例

```yaml
# .quanttide/asset/contract.yaml
spec_version: 0.0.1
version: 1.0.0

assets:
  brd:
    title: 商业需求文档
    type: docs
    path: docs/brd
```

**特点**：
- 声明「应该有」商业需求文档
- 定义资产的基本属性
- 人工维护，版本控制

### Catalog 示例

```yaml
# .quanttide/catalog/2026-04-18T100000Z.yaml（快照文件）
generated_at: 2026-04-18T10:00:00Z
spec_version: 0.0.1
snapshot_reason: "v1.0.0 发布归档"

assets:
  brd:
    title: 商业需求文档
    type: docs
    path: docs/brd
    status: active
    discovered_at: 2026-04-18T09:00:00Z
    last_verified: 2026-04-18T09:30:00Z
    checksum: abc123
    file_count: 5
    total_size: 10240
```

**特点**：
- 记录「特定时间点实际有」商业需求文档
- 包含运行时元数据（状态、时间、校验和等）
- 快照文件纳入版本控制，支持历史追溯

## 使用场景

### 何时关注 Contract

- 项目初始化时定义资产结构
- 约定团队协作规范
- 作为治理的事实源
- 规划「应该有什么」

### 何时关注 Catalog

- **发布归档**：版本发布时记录资产快照
- **合规审计**：提供特定时间点的资产证据
- **漂移检测**：对比历史快照发现资产变更
- **里程碑追溯**：回溯关键节点的资产状态

### 快照触发时机

| 触发条件 | 示例 |
|---------|------|
| 版本发布 | `v1.0.0` 发布时自动生成快照 |
| 定期归档 | 每月/每季度手动触发 |
| 审计需求 | 合规检查前手动触发 |
| 重大变更 | 架构调整前备份快照 |

## 与发现、验证的关系

```
Contract（契约）
    │
    v
Discovery（发现）--> 对比 Contract 与文件系统
    │
    v
Validation（验证）--> 确认资产符合 Contract
    │
    v
Catalog（目录）--> 记录验证通过的资产及元数据
```

## 总结

- **Contract 是「规范」**：定义应该有什么，人工维护，版本控制
- **Catalog 是「快照」**：记录特定时间点实际有什么，自动生成快照，版本控制
- **两者互补**：Contract 驱动治理，Catalog 记录历史，共同支撑审计与追溯
