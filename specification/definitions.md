这个问题的核心在于**职责边界的划分**。在回答之前，我们需要先明确“发现”“注册”“验证”三个动作的精确定义（基于您提供的上下文）：

| 行为 | 定义 |
|------|------|
| **发现（Discovery）** | 从实际环境（文件系统、API、数据库）中识别出存在的资产，并提取其基本属性。 |
| **注册（Registration）** | 将资产的元数据记录到目录中，建立可追溯的身份标识。 |
| **验证（Validation）** | 将发现的资产与契约中的声明进行比对，确认合规性或检测漂移。 |

基于此，分析如下：

---

## 一、三个行为与 Catalog / Contract 的绑定关系分析

### 1. 发现（Discovery）

| 绑定对象 | 理由 | 问题 |
|----------|------|------|
| **Catalog** | 发现的结果直接成为 Catalog 的数据源，Catalog 的快照本质上就是一次发现活动的产物。 | 若 Catalog 与发现工具强耦合，更换扫描方式（如从本地文件改为 API）时 Catalog 结构也会受影响。 |
| **Contract** | 契约定义了“应该发现什么范围”（如 `path: docs/brd`），是发现范围的输入之一。 | 发现本身是事实采集，不应受契约约束（即使没有契约也能发现资产）。 |
| **独立** | 发现作为独立能力，可服务于多种下游（Catalog 快照、实时监控、IDE 插件等）。 | 多一层抽象，增加初期复杂度。 |

**结论**：**发现应与 Catalog 逻辑绑定，但与具体扫描实现解耦**。即 Catalog 模块内部包含“发现调度器”，但具体扫描器可插拔。

### 2. 注册（Registration）

| 绑定对象 | 理由 | 问题 |
|----------|------|------|
| **Catalog** | 注册是 Catalog 的核心写操作，资产只有注册后才在目录中有记录。Catalog 本身就是资产注册表。 | 若注册与 Catalog 紧耦合，则无法支持“预注册”（如在 Contract 中先声明，待资产产生后自动关联）。 |
| **Contract** | 契约可作为注册的“申请单”，即先在契约中声明，再由系统自动或人工确认后注册。 | 契约应是静态规范，不应包含动态注册动作。 |
| **独立** | 注册作为独立服务，统一管理所有资产的 ID 发放与生命周期。 | 对于小型项目过度设计。 |

**结论**：**注册是 Catalog 的固有职责**，但注册的**触发方式**可以与 Contract 联动（如 Contract 变更时提示注册新资产）。架构上，注册方法应归属于 Catalog 模块。

### 3. 验证（Validation）

| 绑定对象 | 理由 | 问题 |
|----------|------|------|
| **Catalog** | 验证通常发生在发现之后，对比发现结果与 Catalog 中上次快照（漂移检测）。 | 验证规则可能来自 Contract，Catalog 自身不持有规则。 |
| **Contract** | 验证的核心是对照契约检查合规性（文件是否存在、字段是否完整）。这是契约执行的自然动作。 | 若验证完全绑定契约，则无法检测“未声明但存在的资产”（幽灵资产）。 |
| **独立** | 验证引擎独立于数据源（Catalog）和规则源（Contract），只做比对逻辑。 | 需要协调两方数据，适合企业级治理平台。 |

**结论**：**验证应与 Contract 绑定**，因为契约是验证规则的唯一真实来源。但验证的执行需要读取 Catalog 的快照数据。

---

## 二、推荐架构：分离关注点，通过协作实现

基于“高内聚、低耦合”原则，三个行为应分布在两个核心模块中，并通过明确的接口协作：

```
┌─────────────────────────────────────────────────────────────┐
│                      资产治理引擎                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────────────┐        ┌──────────────┐                  │
│  │   Contract   │        │   Catalog    │                  │
│  │   (规则源)   │        │   (事实源)   │                  │
│  └──────┬───────┘        └──────┬───────┘                  │
│         │                       │                           │
│         │   ① 声明规范           │   ② 存储快照             │
│         │                       │                           │
│         ▼                       ▼                           │
│  ┌─────────────────────────────────────────┐               │
│  │           行为模块（按需调用）           │               │
│  ├───────────────┬─────────────┬───────────┤               │
│  │   Discovery   │ Registration│ Validation│               │
│  │   (发现)      │  (注册)     │  (验证)   │               │
│  └───────┬───────┴──────┬──────┴─────┬─────┘               │
│          │              │            │                      │
│          ▼              ▼            ▼                      │
│  ┌─────────────────────────────────────────┐               │
│  │          实际资产环境（文件/API/DB）     │               │
│  └─────────────────────────────────────────┘               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 职责分配表

| 模块 | 主要职责 | 包含的行为 |
|------|----------|------------|
| **Contract** | 定义资产规范，版本控制 | 无主动行为，仅提供规范数据 |
| **Catalog** | 存储资产历史快照，提供查询 | **Registration**（写操作）、快照管理 |
| **Discovery** | 扫描实际环境，产出资产清单 | **Discovery**（独立功能，可被 Catalog 调用） |
| **Validation** | 比对规则与事实，生成报告 | **Validation**（独立功能，可被 CI/定时任务调用） |

### 协作流程示例（版本发布场景）

1. **触发**：Git Tag `v1.0.0` 推送
2. **调用 Discovery**：扫描 `docs/` 目录，生成资产清单
3. **调用 Registration**：将清单中的资产记录到 Catalog 的新快照中（自动注册）
4. **调用 Validation**：读取 `contract.yaml`，与刚生成的 Catalog 快照比对
5. **输出**：
   - 验证通过 → 快照归档为 `catalog/2026-04-19-v1.0.0.yaml`
   - 验证失败 → 阻断发布或发出警告

## 三、为什么不应该全部绑定到 Catalog 或 Contract

| 错误绑定方式 | 后果 |
|-------------|------|
| **全部绑定到 Catalog** | Catalog 变得臃肿，包含扫描逻辑、验证规则，失去作为“纯净快照存储”的简单性。 |
| **全部绑定到 Contract** | Contract 变成运行时引擎，而非静态规范，难以版本控制和人工阅读。 |
| **全部独立** | 出现三个孤立的服务，增加集成成本，对中小项目是过度工程。 |

**最佳平衡点**：
- **发现**作为独立工具库，被 Catalog 调用
- **注册**作为 Catalog 的内部方法
- **验证**作为独立函数，被 CI 流程调用，同时依赖 Contract 和 Catalog 的数据

## 四、与您当前 `catalog.py` 的对齐

您的 `catalog.py` 实际上已经隐含了这种分离：

| 代码部分 | 对应的行为 |
|----------|------------|
| `fetch_nodes` + `run_api` | **Discovery**（通过飞书 API 发现资产） |
| `save_json(node_data, ...)` | **Registration**（将资产写入 Catalog 存储） |
| `get_node_detail` | **Enrichment**（获取详情，属于发现的一部分） |
| `metadata.json` 汇总 | **Catalog Index** |

目前缺少的是 **Contract** 和 **Validation**，这两者可以后续以独立模块加入，而无需修改现有 Catalog 结构。

## 五、总结性建议

| 行为 | 应归属/调用关系 | 理由 |
|------|----------------|------|
| **发现** | 独立工具库，被 **Catalog** 调用 | 发现能力可能变化，Catalog 只需知道如何触发 |
| **注册** | **Catalog** 的内部方法 | 注册本质是 Catalog 的写操作 |
| **验证** | 独立模块，读取 **Contract** 和 **Catalog** | 验证规则和事实数据来自两方，引擎应中立 |

**一句话原则**：
> **Catalog 管“记”，Contract 管“约”，发现是“眼睛”，验证是“尺子”。眼睛帮记账，尺子量契约。**
