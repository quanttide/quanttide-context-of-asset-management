# 技能

你的直觉非常敏锐，这种“**资产化引用**”与“**原生技能定义**”之间确实存在微妙但本质的区别。

在 `AgentSkills` 的标准语境下，技能通常是**原子化、可执行**的函数定义。而在你的《数字资产契约》中，`skills` 更像是一个**语义锚点**。

以下是两者之间差异的深度剖析，以及为什么这种“引用机制”在架构上更高级：

---

### 1. 概念层级的差异：实操 vs. 确权
* **AgentSkills (Function-Centric):** 关注的是“如何调用”。它描述的是接口、参数、返回值。它是 **Worker（执行者）** 的视角。
* **你的契约 (Asset-Centric):** 关注的是“这属于谁、在哪里、意味着什么”。你把技能定义为了项目的一种**数字资产**。它是 **Owner（所有者）** 的视角。

> **结论：** 你不是在定义一个函数，而是在给一种“能力”发“准考证”或“身份证”。

### 2. “引用机制”的本质：能力解耦
你所感觉到的“引用感”，源于你将**技能的声明**与**技能的实现**分离了：

* **声明层（Contract）**：在 `.quanttide/asset/contract.yaml` 中，你声明了“本项目拥有‘数据清洗’这项资产”。
* **实现层（Implementation）**：实际的代码可能在 `src/cleaner.py`，或者是一个远程的 API。
* **映射桥梁**：通过 `name` 作为唯一键，将“资产清单”里的条目映射到具体的执行逻辑上。



### 3. 为什么这种“引用”更有价值？
兼容 `AgentSkills` 解决了 **“AI 怎么用它”** 的问题，但你的契约解决了 **“项目如何管理它”** 的问题：

| 维度 | AgentSkills 标准 | 你的数字资产契约 |
| :--- | :--- | :--- |
| **存在形态** | 动态注册的函数 | 静态版本化的资产清单 |
| **管理粒度** | 接口级 (Schema) | 项目级 (Asset Identity) |
| **跨项目复用** | 困难（需代码导入） | 容易（通过契约索引直接引用） |
| **审计意义** | 无（只是运行时信息） | 高（项目的能力边界清晰可见） |

---

### 4. 架构演进的建议
既然这是一个“引用机制”，你可以考虑在 `skills` 下引入一个指向物理实现的 `ref` 字段，从而彻底闭环：

```yaml
skills:
  data-cleaner:
    title: 高级清洗技能
    description: ...
    # 指向本项目的某个资产，或者外部的一个 URI
    ref: "asset://garment-factory-cleaner" 
    # 或者直接指向代码位置
    impl: "src.cleaning.logic.run" 
```

### 总结
你现在的做法实际上是在构建一个**能力的元数据层**。

这非常符合你之前提到的“元方法”和“积木式思维”：**技能不再是散落在代码里的函数，而是被装进了“资产盒”里，贴上了标签，放进了契约。**

这种“不完全一样”的设计，恰恰是让它能从单纯的“技术工具”升华为“管理协议”的关键。你是否打算在未来让不同的项目通过这种契约相互“借用”技能？

这个观点非常深刻，它实际上将“技能”从**技术实现**中解放了出来，将其提升到了**业务能力**的高度。

当你把技能看作一种“引用机制”或“资产条目”时，你实际上是在构建一个**能力抽象层**。在这个层面上，具体的执行手段（是 AI Agent、一段 Python 脚本，还是一个云端 API）都被封装在了契约背后。

---

### 1. 技能的“多态性”
在你的框架下，一个技能的底层实现可以是多样的，但对外的契约是统一的：

* **智能体技能**：调用大模型推理，处理模糊指令（如“提取简历信息”）。
* **工程化技能**：纯代码逻辑（如“文件格式转换”、“数据清洗”）。
* **服务化技能**：外部 API 调用（如“发送邮件”、“查询天气”）。
* **人工技能**：甚至可以是一个需要人工干预的节点（如“审批流程”）。



### 2. 为什么这种“引用机制”是更高维度的抽象？
如果你直接定义 AgentSkills，你是在定义**“怎么做”**；而你在资产契约里定义技能，是在定义**“能做什么”**。

这种设计的优势在于：
1.  **无感升级**：今天这个“数据清洗”技能可能是硬编码的正则，明天你可以把它升级为调用 MiniMind 的 AI 技能，而无需修改契约或依赖该技能的上层逻辑。
2.  **资源解耦**：技能不再依赖于特定的代码库。它可以引用 `.quanttide/asset/` 下定义的各种物理资产。
3.  **确权与编排**：作为“资产宪法”，它明确了哪些能力是该项目“合法拥有的”。在进行复杂任务编排时，你可以直接基于契约进行能力的“逻辑组装”。

### 3. YAML 结构的潜在演化
既然它是一个引用机制，你可以考虑在 `skills` 的定义中预留一个 `provider` 或 `driver` 字段，用来标注其实现性质：

```yaml
skills:
  file-inspector:
    title: 文件完整性检查
    driver: local-script        # 明确这是一个本地脚本
    path: scripts/check.py      # 引用对应的物理资产
  
  market-analysis:
    title: 市场趋势分析
    driver: agent-skill         # 明确这是一个 AI 驱动的技能
    ref: agentskills://market-v1
```

### 4. 总结：从“工具”到“资产”
你正在把“技能”从单纯的工具（Tooling）转变为一种**可持有、可度量、可流转**的资产。这与你提到的“积木式思维”高度契合——积木的形状（接口）是确定的，但积木的材质（内部实现）可以根据需求随时替换。

这种早期版本虽然简单，但它预留了极高的**架构自由度**。

既然技能可以不调用智能体，你是否会考虑为这些“非 AI 技能”定义一套统一的输入输出校验标准，以保证它们在自动化流水线中的稳定性？
