# 业务需求文档（BRD）

本文定义数字资产管理的业务需求，并分析 BRD 与当前 Tutorial 实践之间的代沟。

## BRD 核心定位

BRD 只做一件事：把「要做什么」改造成「发生了什么」。

### 三改变

- 从「要做什么」→「发生了什么」：从结论回到现场
- 从「功能列表」→「业务场景」：系统是行为的组合
- 从「直接给方案」→「延迟决策」：先把问题讲清，再决定怎么做

### 边界

BRD 只描述问题，不描述功能。假设句式是 BRD 到 PRD 的唯一接口：「如果存在这样一个平台，能够……」

## 业务场景

| 场景 | 核心问题 |
|------|----------|
| 数字资产全景图 | 资产关系不透明，管理者看不清全局 |
| AI 交付约束 | 智能体交付质量无法自动验证 |
| 数字资产定价 | 无形资产缺乏客观定价依据 |

## 标准结构

每个场景必须包含：

1. **工作角色**：谁在参与，各自承担什么行为（角色是行为定义，不是岗位定义）
2. **问题**：触发 → 现状 → 困难 → 后果（四要素缺一不可）
3. **假设**：如果存在这样一个平台，能够……

### 问题四要素

问题必须回答四件事：

1. 触发：什么情况下发生
2. 现状：现在是怎么做的
3. 困难：卡在哪里
4. 后果：导致了什么

## 与 Tutorial 的三层代沟

### 第一层：组织 vs 能力

Tutorial 解决了"资产放在哪"——放在不同的 Git 仓库里，每个仓库是一个上下文单元。

BRD 问的是"对这些资产能做什么"——能不能看到它们的关系？能不能自动验收？能不能定价？

Tutorial 建好了货架，BRD 在问能不能安装自动分拣系统、质检设备、估价器。货架是前提，但不是答案。

### 第二层：静态 vs 动态

Tutorial 的方法是静态的：创建仓库、写 SKILL.md、定义目录结构。这是一次性的组织工作。

BRD 的问题是动态的：资产变化了关系要不要更新？智能体每次交付要不要验证？市场价格变了估值要不要调整？

Tutorial 描述的是一套分类法，BRD 需要的是一套工作流。

### 第三层：暗示 vs 强制执行

Tutorial 中智能体的约束来自 `.agents/skills/` 文件。Skill 是**暗示性的**——告诉智能体"你应该这样做"，智能体可以遵守也可以不遵守。

BRD 的 harness 场景要的是**强制性的**——定义规则后系统自动验证，不符合就拒绝通过。

Tutorial 的解法是"给智能体一本操作手册"，BRD 要的是"给交付物装一个安检门"。

## 总结

**Tutorial 解决的是"知识如何组织"，BRD 问的是"组织好的知识如何被自动管理"。**

代沟不是技术上的，是抽象层级上的：Tutorial 在文件系统层，BRD 在应用层。有了好的文件组织方式，不等于自动就有了全景图、质检和定价——这些是需要额外构建的能力。

BRD → PRD → IXD → ADD，QA 验证所有层。
