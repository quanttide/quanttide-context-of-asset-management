# PRD

这是一篇非常精准且深刻的分析。你用“货架 vs 安检门”的比喻，一针见血地指出了当前 AI 工程化实践中最常见的陷阱：把“知识的静态组织”当成了“业务的动态管理”。
你的分析揭示了一个核心矛盾：Tutorial 是为“人（开发者）”设计的，而 BRD 是为“系统（自动化管线）”设计的。 人看到 Git 仓库的分类，能在脑海里自动补全资产关系；但智能体和系统没有这种常识，必须通过额外的工程手段将“暗示”转化为“强制”。
顺着你的三层代沟框架，如果要真正跨越这道代沟，从 Tutorial 的“文件系统层”跃迁到 BRD 的“应用层”，我们需要补齐哪些具体的工程能力？
跨越第一层代沟：从“货架”到“自动分拣与全景图”
*代沟：组织 vs 能力*
Tutorial 把资产散落在各个 Git 仓库（上下文边界）中，这导致了物理隔离。要回答 BRD 的“全景图”问题，需要引入“全局索引层”。
解法： 在 Git 仓库之上构建一个轻量级的“资产图谱”。每次仓库更新时，通过 CI 提取元数据（如 manifest.json），上报到中央图谱数据库（如 Neo4j 或简单的图索引）。
效果： 智能体在执行跨领域任务时，不再是在茫茫 Git 海洋里瞎找，而是先查询图谱，拿到“需要调用哪几个仓库的上下文”的路线图。
跨越第二层代沟：从“分类法”到“事件驱动工作流”
*代沟：静态 vs 动态*
Tutorial 里的 .agents/skills/ 是一张静止的说明书。BRD 需要的是：资产变了，系统跟着动。
解法： 将 Git 仓库的静态结构，接入 CI/CD 的 Webhook 事件流。当一个数字资产（如某个 API 协议、某个数据模型）发生变更时，触发“影响面分析智能体”。
效果： 资产定价模型可以定时重算；关联资产的状态可以从“已发布”自动回滚为“待验证”。这不是靠修改目录结构能实现的，必须依赖工作流引擎。
跨越第三层代沟：从“操作手册”到“安检门”
*代沟：暗示 vs 强制执行*
这是最致命的一层。LLM 本质上是概率模型，Skill 文件是 Prompt（软约束），软约束在规模化交付中一定会发生“漂移”。BRD 的 Harness 场景要求 100% 合规。
解法： 将“怎么做（Skill）”与“怎么做对”解耦。在智能体输出交付物后，必须插入一个确定性的验证层。
结构验证：用 JSON Schema / Pydantic 拦截格式错误。
逻辑验证：用 AST 解析或规则引擎检查是否引用了正确的依赖。
语义验证：用另一个“审查智能体”+ RAG（基于 BRD 强制规则）进行打分。
效果： 智能体可以不遵守 Skill，但只要它不遵守，交付物就通不过“安检门”，会被自动打回。这就是从“人治”走向“法治”。
总结：我们需要一次“架构视角”的升维
你的文章最后一句是点睛之笔。如果用架构语言来翻译你的洞察：
Tutorial 描述的是 Data Layer（数据层/存储层）： 用 Git 管理文件，用 Markdown 管理语义，用目录管理边界。
BRD 诉求的是 Application Layer（应用层/业务层）： 全景图是“查询服务”，质检是“网关服务”，定价是“计算服务”。
没有应用层的数据层，只是死数据。
下一步的实践方向，不应该再是“如何写好一篇 SKILL.md”，而是“如何基于现有的 Git 仓库结构，构建出一套向外暴露 API 的事件驱动架构（EDA）”。让 BRD 里的业务系统，能够通过 API 去调用、验证和计算那些躺在 Git 仓库里的数字资产。
这篇文章非常适合作为团队内部对齐认知的宣言：别再沉溺于搭货架了，咱们该造安检门了。
